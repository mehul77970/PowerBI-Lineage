## [0.11.0] — 2026-04-25 · Drop Mermaid from generated MDs (parked for revival)

User-visible behaviour change: the three Mermaid emission sites in our generated Markdown docs are now gated off. None of `Model.md` §2.4 erDiagram, `Measures.md` per-measure lineage `graph LR`, or `Data Dictionary` per-fact star fragments are written into the MDs. Mermaid revival is parked in [`ROADMAP.md`](../ROADMAP.md#5--mermaid-revival).

### Why

Render-quality across our three target surfaces hasn't been reliable enough to ship Mermaid blocks at scale:

- **GitHub** silently falls back to a plain code block on Mermaid parsing edge-cases. v0.10.2 fixed one (underscore-leader entity names like `_measures`); confidence in the broader grammar surface didn't follow.
- **ADO Wiki** uses Mermaid 8.13.9 — strict-mode parser. We've already worked around `color:` in classDef, `·` middle dots in labels, square brackets in edge labels. There are likely more.
- **Dashboard MD viewer** doesn't render Mermaid at all — it shows the source as a code block. Users wanting visual lineage have the interactive Lineage tab; the Mermaid in MDs was redundant at that surface.

Compounding the inconsistency: a doc emitted to all three targets renders as graphs on GitHub (sometimes), graphs on ADO (sometimes), code blocks on the dashboard (always). Hard to call that a coherent feature.

**Decision:** drop the emission. Re-evaluate when the parsing-quirk count goes to zero across all targets, or shift the visual-lineage investment to the dashboard's interactive Lineage tab (already exists).

### Changed

- **`EMIT_MERMAID` master switch** added to `src/md-generator.ts`, currently `false`. Flipping it back to `true` re-enables all three emission sites without other code changes.
- **All three Mermaid helpers retained** (`mermaidMeasureLineage`, `mermaidTableRelationships`, `mermaidFullModelErDiagram`) — they still compile and would emit valid output. They're just not called.
- **Model.md §2.4 ToC entry** is gated alongside the section emission — Detailed mode no longer includes the entry-relationship-diagram link.
- **README compat table** — Mermaid row updated to ⛔ across all targets with a "parked in ROADMAP" pointer.
- **ROADMAP.md** — new `## 5 · Mermaid revival` theme with two design options (re-enable as-is once grammar is reliable; alternative: drop from MDs permanently and double-down on the dashboard Lineage tab).

### What replaces it

The textual fallbacks in each doc were already pulling weight; the diagrams were the cherry on top. Concretely:

- **Measures.md per-measure block** — now leans on the Depends-on / Used-by chip lists (which became proper anchor links in v0.10.2).
- **Data Dictionary per-fact table** — FK chips already link to their target tables in same-doc anchors (added in v0.10.1 / v0.10.2).
- **Model.md §2.4** — replaced by `### 2.3 Relationship inventory` (already there: a sortable table of every relationship with cardinality + filter direction).

### Tests

- **`tests/md-mermaid.test.ts` rewritten.** Was: assert ≥5 blocks in Measures.md, ≥3 star fragments in Data Dictionary. Now: assert **zero** Mermaid blocks anywhere across the three docs. The originals are summarised in the file's docstring so the revival path is documented.
- **`tests/md-modes.test.ts` updated.** Two assertions involving `erDiagram` substring presence in Model.md flipped to assert absence.
- **267/267 pass** (was 270 in v0.10.2 — three Mermaid-presence tests retired).

- `npm test` — 267/267 green
- `npm run build:browser` — bundle 304 KB (under 400 KB cap)
